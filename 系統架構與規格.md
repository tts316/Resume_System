## 2. 資料庫架構

- **正式資料庫：Cloud SQL PostgreSQL 16**（2026-07-10 完成從 Google Sheets 搬遷並切換上線）
  - 個體 `resume-pg` / DB `resume` / app 帳號 `resume_app`
  - Connection name：`procuresys-499802:asia-east1:resume-pg`
  - Secret：`resume-pg-app-password`、`resume-pg-root-password`
  - 環境變數 `DB_BACKEND=postgres` 啟用 PG 路徑；未設/`=sheets` 則回退用 Google Sheets（雙後端相容層仍保留，理論上可回退）
- **雙後端轉接層**（`app.py` 內 `PGBackend`/`PGWorksheet`/`PGSpreadsheet`）：模擬 gspread worksheet 介面，讓商業邏輯不用因換 DB 而改寫。核心樞紐是每表 `_rn BIGSERIAL` 欄記錄列序（PG 無天然列序）。
- **欄位自癒機制**：`PGBackend.__init__` 啟動時對既有表跑 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`，新增欄位（如 `signature`、`docs_enabled`）不需手動改 schema，服務重啟自動補齊。
- 三張表：
  - `users`：email(PK) / password / name / role(admin\|pm\|candidate) / creator_email / created_at
  - `resumes`：email(PK) / status(New\|Draft\|Submitted\|Returned\|Approved) / 95+ 欄履歷欄位（見第 4 節）+ `signature` / `signed_at` / `docs_enabled` / `docs_submitted_at`
  - `system_settings`：key(PK) / value（目前僅存 `logo` 的 base64 圖檔字串）
  - `onboarding_docs`（新增資料表，非既有 3 表之一）：id / email / category / slot / filename / mime / data(bytea) / created_at，索引 `idx_onboarding_email`
- **效能優化**：
  - `load_df()` 包 `@st.cache_data(ttl=30)`，寫入後 `_invalidate_cache()` 立即清快取
  - PDF 產生包 `@st.cache_data(ttl=600)`，依整列內容當 key
  - `save_resume`/`hr_update_status` 改批次寫入（單條多欄 `UPDATE`），存一筆履歷從 ~95 條 SQL 降為 1 條

參考手冊：`C:\Users\tonytsai\.claude\projects\C---Resume-System\memory\pg-migration-guide.md`（Sheets→PG 搬遷可複用流程）

## 3. 角色與權限

| 角色 | 說明 | 權限範圍 |
|---|---|---|
| `admin` | 人資主管 | 全部功能；唯一能「建立人資 PM」帳號的角色 |
| `pm` | 人資 PM | 只能「邀請面試者」（不能建立其他 PM）；其餘分頁（履歷審核/表單管理/到職文件管理）**看得到**，但資料範圍**限縮在自己邀請的候選人** |
| `candidate` | 求職者/面試者 | 履歷填寫 + 履歷查詢/確認 + 到職文件（若 PM/admin 開放） |

> **2026-07-20 明確決策**：PM 權限採「保守解讀」——即只有帳號**建立**權限限制在 admin，其他既有分頁 PM 仍可見，但資料以 `creator_email == PM信箱` 過濾。此為最終決策，非待確認事項。

登入去空格防呆：帳號/密碼比對前雙邊 `.strip()`，另有 autofill 診斷訊息（顯示收到帳號/密碼碼數，協助排查瀏覽器自動填寫失敗的狀況）。

## 4. 後台（admin/pm）— `admin_page()`

Tabs 順序：`📧 發送邀請` → `📋 履歷審核` → `📊 表單管理` → `📁 到職文件管理` → `⚙️ 設定`（僅 admin 可見）

### 4.1 發送邀請
- Radio 切換「邀請面試者」/「建立人資 PM」（PM 帳號看不到後者，因為建立 PM 僅限 admin）
- **邀請面試者**：`st.data_editor` 批次表格，一次最多 6 列（姓名/Email/履歷類型/**面試單位**），按「發送面試邀請」一次批次寫入帳號＋寄邀請信；空白列自動略過；每列 4 欄位都必填才會發送，缺欄逐列標示；**並檢核面試單位與履歷類型是否相符**（總公司↔總公司單位、分公司↔分公司），不符逐列標示不發送；完成後顯示逐列 ✅/❌ 摘要
- **面試單位**選項來自「⚙️ 設定 → 公司組織維護」，寫入 `resumes.interview_unit`；分公司類型會一併把「區域 / 分公司」帶入求職者的 `branch_region`/`branch_location`（求職者仍可自行調整志願），並顯示於邀請 email 與履歷 PDF
- 邀請信為 HTML 格式（logo、按鈕式連結），純文字 fallback

### 4.2 履歷審核
- 依 status 篩選待審/已核可/已退件；PM 只看自己邀請的
- 每筆可下載完整 PDF、AI 履歷分析（呼叫 `claude-haiku-4-5`，需設定 `ANTHROPIC_API_KEY`）
- 核准需填面試資訊（日期/時間/地點/單位/主管/備註），核准後 email 通知求職者
- 退件需填退件原因，email 通知求職者修改重送

### 4.3 表單管理
- 起訖月份查詢（預設近 2 個月，避免畫面月份過多）
- 每筆「操作」欄同一區塊：
  - ☑ 開放到職文件（`docs_enabled`，即時生效、無需重整）
  - 📤 提醒上傳（僅在已勾選開放到職文件時可按，未勾選反灰；寄「提醒您上傳到職文件」email）
  - 催促填寫/催促修改（原有功能，button key 含列索引避免 email 重複時 key 衝突崩潰）
  - **匯入管理系統**（僅在已開放到職文件時出現）：「求職者編號」輸入框（PM 於管理系統查詢後填入）＋「📥 匯入管理系統」按鈕。以求職者編號為識別 id，將人員資料與到職文件匯入聯成電腦管理系統。**⚠️ 目前為預留接口**：管理系統匯入 API 技術文件尚未提供，按下僅儲存求職者編號（存 `resumes.mgmt_cand_no`）；待文件補上後於 `_mgmt_import()` 組裝 payload 呼叫（建議 URL/Token 存 `system_settings` 的 `mgmt_import_url`/`mgmt_import_token`）
- 求職者帳號刪除：多選 checkbox，**僅未勾選「開放到職文件」的候選人**可刪除（已開放者顯示 🔒 鎖定，避免刪掉正在走到職文件流程的人）；刪除前二次確認、刪除後顯示成功/失敗摘要
  - admin 可刪全部；PM 只能刪自己邀請的

### 4.4 到職文件管理（新分頁，2026-07 新增）
- 僅列出 **status=Approved** 的求職者（admin 全部／PM 限自己邀請的）
- 可查閱調閱求職者已上傳的到職文件（延遲載入，按調閱才讀 bytea）
- 可填寫補送說明，按下發送後 email 通知求職者補件
- 提示文字：「開啟上傳權限請至『表單管理』」（權限開關與文件管理分離，符合「表單管理負責開關、到職文件管理負責後續操作」的分工）

### 4.6 人員管理（僅 admin，設定右側）
- 列出所有 `admin` / `pm` 帳號，逐人維護：**姓名 / 員工編號 / 單位（選單，來源同組織維護）/ 重設密碼**；Email 為主鍵不可修改
- **員工編號**：串接聯成電腦管理系統「待辦通知 API」的識別 ID（UserId），限數字
- **PM 離職**：必須同時「指定接手 PM」（只列在職 PM）才能執行 → 彈出確認對話視窗（顯示將轉移幾位求職者）→ 確認後把該 PM 經手的所有求職者 `creator_email` 改為接手 PM、並標記 `active=N` → 完成後顯示摘要
- **admin 帳號**：可修改資料，但**不提供離職設定、不可刪除**
- `users` 新增欄位 `emp_id` / `unit` / `active`（自癒建欄，`active` 預設 `Y`；空字串一律視為在職）

### 4.5 設定（僅 admin）
- Logo 上傳：存 DB `system_settings.logo`（base64），非寫死 URL。讀取統一走 `_logo_src()` 共用函式，正確處理是否已含 `data:` 前綴，避免重複前綴造成圖片壞掉。
- **待辦通知 API 設定**：維護發送/取消的 URL 與 Token（見 §6.1），存 `system_settings`。
- **公司組織維護**（`org_units` 表，可新增/編輯/刪除列後儲存）：
  - **總公司**：3 層由上而下 `群 / 部 / 處`；「處」可直屬群（「部」留空）
  - **分公司**：3 層由上而下 `群 / 區域 / 分公司`
  - **總公司獨立單位**：總經理室、董事長室（僅單位名稱）
  - 首次啟動自動灌入預設值（7 部＋4 群＋2 獨立單位＋既有 6 區域 24 分校），之後完全由 admin 維護、不再覆寫
  - 維護結果即為「發送邀請」的面試單位選項來源

## 5. 候選人頁（求職者）— `candidate_page()`

三個分頁：`📝 履歷填寫` → `🖋️ 履歷查詢/確認` → `📎 到職文件`（第三分頁僅 `docs_enabled=Y` 時開放操作，否則顯示未開放提示）

### 5.1 履歷填寫（`_render_fill`）
- 步驟進度條（①基本資料 ②學歷 ③工作經歷 ④其他資訊 ⑤確認送出）
- 必填欄位標示（姓名*/手機*/地址*/學歷1*）
- 學歷區塊採可展開/收合（比照工作經歷樣式）：學歷1 預設展開＋標必填，學歷2/3 預設收合
- 工作經歷最近 4 筆，各自 expander
- 基本資料：**身高/體重欄位已移除**（2026-07-22，不再填寫；DB 欄位保留不刪，舊資料仍在）
- **星座**：位於血型旁，**依生日自動對應**、唯讀不需填寫（`_zodiac_of()` 計算，不存 DB，PDF/審核頁一併顯示）。受 `st.form` 限制，改生日後需暫存/送出才會刷新
- **所有年月欄位改為「年」「月」下拉選單**（學歷入學/畢業、經歷起訖共 14 處），取代原本手打 YYYY/MM；下拉可直接輸入文字篩選。生日維持 `st.date_input`（日曆點選＋手動輸入）
- **語言能力**：最多 3 列，「語言」選單（15 種）＋「能力等級」選單（優/普通/略通），存 `lang_1..3` / `lang_1..3_level`
- 分公司（Branch）專屬：任職區域/分校/輪調意願/輪班意願等問卷
- 技能與自傳區：
  - 新增欄位「您找工作最重視的3個條件為?(300字內)」（`top3_conditions`）
  - 追蹤職缺 QR code 下方：紅底白字醒目警語（黑名單條款＋資料真實性條款，逐字比照公司規定文字）
- 送出/暫存前端＋後端雙重驗證（必填、YYYY/MM 格式、數字欄位、英文姓名格式等）
- Approved 狀態顯示恭喜訊息＋面試資訊；**`st.balloons()` 氣球動畫僅在該次登入 session 第一次看到 Approved 狀態時放一次**（`session_state['_balloons_shown']` 旗標控制，避免簽名頁互動每次 rerun 都重播）

### 5.2 履歷查詢/確認（`_render_confirm`，2026-07 新增）
- **履歷調閱**：已送出（Submitted/Approved/Returned）可下載目前狀態的完整 PDF
- **手寫簽名**（僅 `status==Approved` 後開放，確保履歷內容已審核完成才可簽）：
  1. Email 驗證碼閘門：按鈕寄 6 位數碼到本人信箱，`session_state` 記錄到期時間，**5 分鐘內**有效，需輸入正確才解鎖
  2. 簽名方框：`streamlit-drawable-canvas`，手機手指／電腦滑鼠或手寫板皆可操作
  3. 確認後存 `resumes.signature`（base64 PNG）＋`signed_at`，PDF 產生時自動套印於簽名欄（無簽名則維持原底線空格）

### 5.3 到職文件（`_render_docs`，2026-07 新增）
- 前提：PM/admin 於「表單管理」勾選「開放到職文件」後才能操作；未開放顯示提示訊息
- **簽名履歷（手寫履歷）**：不需另外上傳，於到職文件頁頂直接查閱/下載系統內含簽名的履歷 PDF；若尚未簽名則顯示提醒，引導至「履歷查詢/確認」完成簽名
- 上傳類別（6 類，各最多 3 檔）：**人力銀行履歷\***、**身分證正反面\***、**最高學歷證書\***、良民證、勞保明細、退伍令。**前三項（標 \*）為必要文件**
- **送出檢核**：缺任一必要文件會顯示提醒並**擋下送出**，列出缺哪幾項，要求完整上傳後才能送出
- UI：頂部必要文件完成進度條、每類以 ✅/🔴/⬜ 狀態徽章＋卡片式容器；PM 端「到職文件管理」列頭顯示「必要 X/3」完成度
- 格式限制：PDF / JPG / PNG / DOC；單檔 5MB 上限（`--server.maxUploadSize=5`，上傳框直接顯示 5MB）
- 可刪除已上傳文件、重新上傳
- 送出：按下「送出」後 email 通知該求職者的建立 PM（`creator_email`），之後仍可補傳/調閱/刪除（送出不鎖定）
- 檔案存 `onboarding_docs`（bytea），下載採延遲載入避免每次 rerun 讀取全部檔案內容

### 6.1 待辦通知 API 串接（聯成電腦管理系統）
串接管理系統待辦 API，於下列事件對 PM 發送「待辦通知」（Type=2 任務），並在對應動作完成時自動取消：

| 建立時機 | 收件 PM | 事件碼 | 取消時機 |
|---|---|---|---|
| 求職者送出履歷待審 | 建立該候選人的 PM | `submit` | PM 核准或退件時 |
| 求職者完成履歷簽名 | 同上 | `sign` | PM 於表單管理**開啟到職文件**時 |
| 求職者送出到職文件 | 同上 | `docs` | PM 於到職文件管理**調閱**時 |

- API：建立（`POST .../Todo`，回傳 `TodoId`）、取消（`POST .../Todo/Cancel`，帶 `TodoId`）
- `UserId` = 該 PM 在「人員管理」設定的**員工編號**（數字）；PM 無員工編號則略過
- `TodoId` 記於 `todo_refs` 表（`cand_email`+`event` 為 PK），供後續取消
- **URL 與 Token 存 PG（`system_settings`），由 admin 於「⚙️ 設定 → 🔔 待辦通知 API 設定」維護**：4 個鍵 `todo_create_url`/`todo_create_token`（發送）、`todo_cancel_url`/`todo_cancel_token`（取消）。Token 欄以 password 輸入、留空＝不變更、畫面只顯示「已設定/未設定」不回顯明碼。任一缺（URL 或 Token）→ 該動作靜默略過。所有 API 呼叫 try/except、逾時 8 秒，失敗不影響 email 與使用者操作
- **待辦連結可直接登入**：Link 帶 `?lt=<token>`，token 為 HMAC-SHA256 簽章（含 14 天效期，密鑰 env `AUTO_LOGIN_SECRET`）。PM 點連結 → 系統驗章通過即免帳密登入、並清除網址上的 token。未設 `AUTO_LOGIN_SECRET` 則連結退化為一般登入頁（需自行登入）。已離職帳號不予自動登入。
